# 前端可视性与操作完整性 — 多角度交叉验证最终评审

**评判标准**：图标/按钮/下拉框/状态标签/提示文案统一视为"可视操作元素"  
**方法**：5 个独立角度交叉验证  
**日期**：2026-05-15  
**产出**：12 条新业务需求

---

## 方法说明

5 个独立角度逐一审计，每个角度独立产出发现，最后交叉取最低分：

| 角度 | 审计内容 | 评判尺度 |
|------|---------|---------|
| **A. 任务闭环** | 用户能否仅通过 UI 完成一个完整任务（触发→进度→结果→失败排查→重试） | 闭环/断裂 |
| **B. 信息可见** | 后端返回的每个字段，在用户需要看到的时刻，是否可见 | 可见/隐藏 |
| **C. 操作可达** | 后端支持的每个操作，前端是否有对应触发元素 | 可达/缺失 |
| **D. 状态可辨** | 每条数据在不同状态下，用户能否一眼识别当前状态和下一步操作 | 可辨/混淆 |
| **E. 错误可恢复** | 每种失败场景下，用户能否在 UI 内完成诊断→修复→重试 | 可恢复/阻塞 |

## 角度 B：信息可见性审计（关键后端字段）

逐字段验证：后端返回了 → 前端存了 → 用户看到了 → 用户看懂了。

| 后端字段 | 存了 | 看到 | 看懂 | 差距 |
|---------|------|------|------|------|
| `results[].error`（批量提交逐条失败原因） | ❌ | ❌ | ❌ | 仅渲染计数 |
| `results[].alpha_id`（哪个 Alpha 失败） | ❌ | ❌ | ❌ | 同上 |
| `checks[].detail`（检查失败具体原因） | ✅ | ✅ | ❌ | 技术名无翻译 |
| `requires_official_check`（待官方检查） | ✅ | ❌ | ❌ | 无视觉提示 |
| `cloud_correlation_risk.max_similarity` | ✅ | ❌ | ❌ | 仅用 level |
| `cloud_correlation_risk.matched_alpha_id` | ✅ | ❌ | ❌ | 完全隐藏 |
| `cloud_status.match`（匹配方式） | ✅ | ❌ | ❌ | 完全隐藏 |
| `submission`（提交结果解析） | ✅ | ⚠ | ❌ | 原始 JSON |
| `profile.username`（当前账户名） | ✅ | ❌ | ❌ | 无显示 |
| `lifecycle stage=blocked` | ✅ | ❌ | ❌ | 未匹配正则 |
| `health.status` | ❌ | ❌ | ❌ | API 未调用 |

**结论**：11 个关键字段中，6 个存了但用户看不到，2 个存了但用户看不懂，3 个完全缺失。

## 角度 D：状态可辨性审计（用户一眼识别）

验证每种数据状态下用户能否清楚知道"这是什么状态"→"下一步该做什么"。

| 数据状态 | 当前视觉 | 当前名称 | 用户可辨？ | 有下一步指引？ |
|---------|---------|---------|-----------|--------------|
| 候选池 | teal 卡片 | "候选池" | ✅ | ✅ 自动进回测 |
| 等待回测 | stat 卡片 | "等待回测" | ✅ | ✅ 排序分优先 |
| 回测中 | 进度条+槽位 | "回测中" | ✅ | ✅ 轮询中 |
| 达标 | green 卡片 | "达标" | ✅ | ✅ 提示执行检查 |
| 可提交 | green 卡片 | "可提交" | ✅ | ✅ 勾选提交 |
| 已提交 | stat 卡片 | "已提交" | ✅ | ✅ |
| 不达标 | red 卡片 | "不达标" | ✅ | ⚠ 回溯但无指引 |
| **不达标 → BLOCKED** | 不存在 | 不存在 | ❌ | ❌ |
| 检查未通过 | red 标记 | "production_gate" | ❌ | ❌ |
| 需复检 | 文字 | "需复检" | ✅ | ⚠ 知道需复检 |
| **待官方检查** | 无 | 无 | ❌ | ❌ |
| 批量提交部分失败 | 文字 | "成功 X 失败 Y" | ❌ | ❌ |

**结论**：12 种状态中 4 种不可辨（BLOCKED / 检查失败原因 / 待官方检查 / 批量部分失败）。

## 角度 E：错误恢复审计（用户能否在 UI 内自救）

| 失败场景 | 当前提示 | 诊断信息 | 修复路径 | 重试入口 | 可恢复？ |
|---------|---------|---------|---------|---------|---------|
| 认证失败 | Toast "认证失败" | ⚠ 原因模糊 | 修改凭据 | ✅ 重新连接 | ⚠ |
| 云端同步失败 | Toast | ❌ 无明细 | 未知 | ✅ | ❌ |
| 生产异常 | Toast + 状态 | ⚠ 部分 | 修改参数 | ✅ | ⚠ |
| 单条检查未通过 | 详情面板 | ❌ 技术名 | ❌ 无指引 | ✅ 重新检查 | ❌ |
| 批量检查有失败 | 详情面板 | ❌ 技术名 | ❌ 无指引 | ✅ | ❌ |
| 单条提交失败 | Toast | ❌ 原始JSON | ❌ | ❌ | ❌ |
| 批量提交部分失败 | "成功X失败Y" | ❌ | ❌ | ❌ | ❌ |
| 提交被阻断 | ❌ 找不到 | ❌ | ❌ | ❌ | ❌ |
| 检查过期 | 文字 "需复检" | ✅ | ✅ 重新检查 | ✅ | ✅ |
| Alpha 相似风险 | attention() | ⚠ 仅level | ❌ | ❌ | ❌ |

**结论**：10 种失败场景中仅 1 种可自主恢复（检查过期），6 种无任何恢复路径。

## 最终产出：12 条新业务需求

按交叉验证严重度（❌ 数量）排序：

### P0 — 5 角度全 ❌（上线阻断）

**BR-01：批量提交失败逐条明细与重试**

| 项目 | 内容 |
|------|------|
| 触发 | 批量提交返回后，若 `failed > 0` 则在结果区域自动展开 |
| 可视元素 | 展开面板 + 失败列表（每行：Alpha ID + 失败原因 + `[单条重试]` 按钮）+ `[全部重试失败项]` 按钮 |
| 数据来源 | `POST /api/submit_batch` → `results[]` 数组 |
| 验收标准 | 全部失败时用户仍能看到每个 Alpha 的失败原因；可逐条或批量重试 |

**BR-02：BLOCKED 状态在失败/不达标视图中可见**

| 项目 | 内容 |
|------|------|
| 触发 | `failedRows()` 匹配 `blocked` 状态的生命周期记录 |
| 可视元素 | 失败视图增加"阻断"子标签 + 阻断原因列（来自 `submission_blocked` 记录的 `note` 字段） |
| 数据来源 | `GET /api/lifecycle` → `records[]` 中 `stage=submission_blocked` 的记录 |
| 验收标准 | 提交被阻断的 Alpha 在"不达标"页卡中可见，附阻断原因 |

**BR-03：检查失败原因全量中文化 + 操作指引**

| 项目 | 内容 |
|------|------|
| 触发 | 每次检查结果渲染时 |
| 可视元素 | 每个检查项显示中文名称 + `detail` 字段描述 + 操作建议（如"减少与其他 Alpha 的相关性"） |
| 数据来源 | `checks[].name` → `humanCheckName()` 映射表补全至 25 项 |
| 验收标准 | 零技术名直接暴露给用户；每个失败检查项有对应操作建议 |

**BR-04：检查结果跨刷新持久化恢复**

| 项目 | 内容 |
|------|------|
| 触发 | 页面初始化 + 在可提交/达标视图渲染时 |
| 可视元素 | 页面加载时自动恢复上次检查状态，`checkResults` 重新填充 |
| 数据来源 | 新增 `GET /api/check_results` → 读取 `checks.jsonl` |
| 验收标准 | 刷新后"可提交"视图与刷新前一致；过期检查标记为"需复检" |

### P1 — 3+ 角度 ❌（QA 阶段）

**BR-05：Alpha Type 下拉完整化**

| 项目 | 内容 |
|------|------|

| 验收标准 | 4 个 Alpha Type 可切换，提交时正确传递到 settings.type |

**BR-06：云端相似风险详情面板**

| 项目 | 内容 |
|------|------|
| 触发 | 查看 Alpha 详情时，若 `correlation_risk.level !== "low"` |
| 可视元素 | 详情面板中展开相似风险区域：相似度百分比 + 匹配 Alpha ID + 匹配 Alpha 状态 + 图标（🔗 表达式匹配 / 📝 官方 ID 匹配） |
| 数据来源 | `cloud_correlation_risk`（含 `max_similarity`, `matched_alpha_id`, `matched_status`） |
| 验收标准 | 用户能看到"和哪个 Alpha 有相似风险、相似度是多少、该 Alpha 是否已提交" |

**BR-07：待官方检查状态标识**

| 项目 | 内容 |
|------|------|
| 触发 | `requires_official_check === true` 时 |
| 可视元素 | 状态列显示橙色 `⚠ 待官方检查` 标签 + 提示"需执行全部检查模式包含官方预提交检查" |
| 数据来源 | `check.requires_official_check` 字段 |
| 验收标准 | 用户能区分"本地通过但需官方复查"与"完全通过可提交" |

**BR-08：提交结果解析展示**

| 项目 | 内容 |
|------|------|
| 触发 | 提交成功返回后 |
| 可视元素 | 解析 `submission` 字段显示：官方 Alpha ID + 提交状态 + 时间；失败则显示具体错误 |
| 数据来源 | `submission` dict |
| 验收标准 | 提交后用户看到可读的确认信息而非原始 JSON |

**BR-09：服务状态指示灯**

| 项目 | 内容 |
|------|------|
| 触发 | 页面加载 + 定时轮询 |
| 可视元素 | `header` 右侧 🟢 在线 / 🔴 离线 指示灯 + 最近检查时间 |
| 数据来源 | `GET /api/health` |
| 验收标准 | 服务断连时用户能在 5 秒内感知 |

**BR-10：退出登录按钮**

| 项目 | 内容 |
|------|------|
| 可视元素 | `header` 用户区域下拉 `[退出登录]` 按钮 |
| 数据来源 | `POST /api/logout` |
| 验收标准 | 点击后清除 session + 回到登录状态 |
