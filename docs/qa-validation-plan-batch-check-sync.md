## 4. 必须验证的准入项

以下为**上线准入必测项**，根据验证结果决定是否阻塞上线：

### 4.1 TC-014：fields/operators 刷新失败但 UI 或 summary 显示全部同步成功
- **问题描述**：最严重的误导用户问题，会导致用户误以为上下文已完整同步
- **代码位置**：`web.py` 第965-975行（`updateProgress` 函数处理 `context` 阶段）
- **验证方法**：模拟 fields 同步失败，检查 UI 是否显示 warning 而非"成功"
- **判定标准**：
  - **阻塞上线**：如果验证发现 UI/summary 在同步失败时仍显示"全部同步成功"
  - **不阻塞**：如果已有 warning 或状态提示，用户能识别上下文不是最新

### 4.2 TC-018：使用旧上下文但没有任何提示
- **问题描述**：当使用缓存或默认值时，必须明确提示用户
- **代码位置**：`pipeline.py` 第483-490行（fallback 逻辑）
- **验证方法**：模拟缓存/默认值场景，检查 UI 提示
- **判定标准**：
  - **阻塞上线**：如果使用旧上下文没有任何提示
  - **不阻塞**：如果已有 warning 或状态提示

### 4.3 TC-015：认证失败但仍显示同步成功
- **问题描述**：认证是前提，认证失败不应显示任何同步成功
- **代码位置**：`pipeline.py` 第141行（`authenticate()` 调用）
- **验证方法**：使用无效凭据，检查 UI 状态
- **判定标准**：
  - **阻塞上线**：如果认证失败后仍显示同步成功
  - **不阻塞**：如果明确提示认证失败

### 4.4 TC-002、TC-003、TC-004：同步失败导致批量检查崩溃
- **问题描述**：首期允许 best-effort 降级，但不应崩溃
- **代码位置**：`pipeline.py` 第469-482行（异常处理逻辑）
- **验证方法**：模拟各种同步失败，检查是否崩溃，错误信息是否可理解
- **判定标准**：
  - **阻塞上线**：如果同步失败导致崩溃且错误信息不可理解
  - **不阻塞**：如果优雅降级，有可理解的错误信息

### 4.5 TC-001 相关：云端 Alpha 同步失败但候选仍被错误判断为可提交
- **问题描述**：云端 Alpha 状态影响提交判断，同步失败可能导致错误提交
- **代码位置**：`pipeline.py` 第142行（`_sync_cloud_alphas()` 调用）
- **验证方法**：模拟云端同步失败，检查候选提交判断逻辑
- **判定标准**：
  - **阻塞上线**：如果云端 Alpha 同步失败后，候选仍被错误判断为可提交
  - **不阻塞**：如果正确处理同步失败，不影响提交判断

## 5. 可进入 Next Sprint 的问题

以下问题**不阻塞首期上线**，但需要后续优化：

### 5.1 summary 未展示 fields_count / operators_count
- **当前状态**：代码已包含（pipeline.py 第493-494行），但 UI 展示可能不完整
- **优化建议**：确保 Web UI 的 `renderInsight()` 函数展示这两个数值
- **优先级**：中

### 5.2 warning 文案不够产品化
- **当前状态**：warning 是技术语言（如"官方上下文接口触发限流"）
- **优化建议**：需要产品化文案，让用户更容易理解（如"字段同步失败，使用基础字段继续"）
- **优先级**：中
- **需产品确认**：warning 文案的产品化表述

### 5.3 无法展示使用缓存的具体时间
- **当前状态**：缓存逻辑不记录缓存时间（`official.py` 的 `_read_cache()` 和 `_write_cache()`）
- **优化建议**：在缓存中添加时间戳，并在 UI 展示（如"使用 2 小时前的缓存数据"）
- **优先级**：低

### 5.4 无法区分字段缓存和算子缓存来源
- **当前状态**：`context_summary.source` 是 "official_api_or_cache"，不够细化
- **优化建议**：细化 source 字段，区分 fields 和 operators 的来源（如 "fields: cache, operators: default"）
- **优先级**：中

### 5.5 云端 Alpha 同步失败时批量检查仍继续执行
- **当前状态**：代码允许失败后继续执行（pipeline.py 第356-369行）
- **需产品确认**：是否允许失败后继续，还是应该终止
- **优先级**：低

### 5.6 403/401 权限失败的处理策略
- **当前状态**：代码对 403/401 和 429/timeout 的处理方式不同（代码第470-482行）
- **需产品确认**：403/401 是否允许 best-effort 降级，还是应终止批量检查
- **优先级**：高（需在首期前确认）

8. **缓存过期**：模拟缓存存在但过期

- 创建 stub API 类，控制 `list_fields()`、`list_operators()`、`list_user_alphas()` 的返回
- 验证 `context_summary` 是否正确
- 验证 UI 是否显示正确的状态
