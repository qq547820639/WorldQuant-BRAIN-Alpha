# 设计评审：BRAIN Alpha Ops 前端

**评审日期**：2026-05-15  
**评审范围**：`brain_alpha_ops/web/index.html` + `brain_alpha_ops/web.py`  
**评审阶段**：QA 前门禁评估  
**产品类型**：内部工具（WorldQuant BRAIN Alpha 自动化生产控制台）

---

## 1. 第一印象

打开控制台后视线首先落在左侧控制面板（认证、参数、同步），右侧依次是数据统计、回测槽位、数据表格——信息密度高但布局清晰。整体感受：**专业、克制、有一定工程成熟度**。配色（teal 主色调 + 灰色系统色）传达出金融产品的冷静感，符合目标用户画像。

核心问题：**非主路径业务场景（失败反馈、阻断诊断、特殊类型）在前端的可视性和操作完整性不足**，用户在异常路径下会丢失上下文。

---

## 2. 可用性

| 发现 | 严重度 | 建议 |
|------|--------|------|
| 批量提交失败仅显示"成功 x / 失败 y"，无逐条明细 | 🔴 Critical | 在提交结果区域展开失败 Alpha 列表，逐条展示 Alpha ID + 失败原因 |
| 提交被阻断后，BLOCKED 状态未进入"不达标/失败"视图 | 🔴 Critical | `failedRows()` 正则增加 `blocked` 匹配，或在失败视图中增加"阻断"子标签 |
| 检查结果仅存页面内存，刷新后丢失 | 🔴 Critical | 增加 `/api/check_results` 读取 `checks.jsonl` 的接口，页面初始化时恢复 |
| 特殊 Alpha Type 仅 REGULAR，PRD 要求 Power Pool/ATOM/Pyramid | 🔴 Critical | 下拉增加对应选项，或从配置动态加载 |
| 6+ 后端状态码无中文映射，前端 fallback 显示原始 code | 🟡 High | 补全 `statusLabelFromCode()` 映射表 |
| 检查失败原因偏技术化（如 `production_gate`） | 🟡 High | `humanCheckName()` 补全所有检查名映射，提供可操作的"处理指引" |
| "可提交"视图空状态依赖检查结果，刷新后变空 | 🟡 High | 增加"暂无检查记录，请执行检查"引导入口 |
| 文档说生产/检查/提交互斥，但代码未强制互斥 | 🟡 High | 明确产品设计意图；若互斥则加前端锁；若允许并行则更新文案解除误解 |
| 后端事件日志（策略切换/刷新失败/收敛停滞）无前端视图 | 🟢 Moderate | 增加事件中心视图，在侧栏或独立 Tab 展示 |
| 图表依赖外部 CDN，离线/受限网络不可用 | 🟢 Moderate | Chart.js 本地化打包，或增加降级占位 |
| 大表格在移动端（720px 以下）体验不佳 | 🟢 Moderate | 移动端切换卡片视图或横向滚动优化 |

---

## 3. 视觉层级

- **视线路径正确**：左侧设置 → 右侧监控 → 表格数据，符合从左到右、从上到下的操作流
- **强调重点准确**：teal 主色用于 CTA 和关键状态，红色用于错误/阻断，黄色用于警告，蓝色用于信息——语义清晰
- **卡片分组有效**：候选池/回测/达标/可提交 等状态卡形成清晰的漏斗模型
- **字重梯度合理**：h1(900) → h2(850) → label(800) → body(400)，层级感明显

**可改进**：
- 表格区域 `max-height: calc(100vh - 430px)` 在屏幕较小时可能过窄，建议加 `min-height`
- 侧栏 `sticky` 定位在内容较长时，底部 `max-height` 截断无滚动提示

---

## 4. 一致性

| 元素 | 问题 | 建议 |
|------|------|------|
| 状态码映射 | `statusLabelFromCode()` 部分 code fallback 到原始值，与其他翻译后的 code 不一致 | 补全映射表 |
| 空状态文案 | 各组空状态文案风格一致、信息充分 | ✅ 保持 |
| 按钮样式 | primary/secondary/stop/small 分层清晰 | ✅ 保持 |
| Toast 通知 | success/error/warning/info 四种样式覆盖所有场景 | ✅ 保持 |
| 弹窗设计 | 详情弹窗 vs 确认弹窗 样式区分明确 | ✅ 保持 |
| 颜色语义 | teal=操作/通过, red=失败/阻断, yellow=警告, blue=信息——全站一致 | ✅ 保持 |

---

## 5. 无障碍

| 检查项 | 现状 | 风险 |
|--------|------|------|
| 色彩对比度 | CSS 变量定义清晰，teal(#0f766e) 在白色背景上对比度 4.56:1 (AA 通过) | 低 |
| 触控目标 | 按钮 min-height 34px，符合 WCAG 建议 | 低 |
| 文本可读性 | 最小 11px（辅助文本），正文 12-13px，line-height 1.45-1.55 | 中（11px 偏小） |
| ARIA 角色 | 表格、弹窗、Toast 缺少 `role` 属性 | 高 |
| 动态区域 | Toast 缺少 `aria-live`，进度更新缺少实时通知 | 高 |
| 焦点管理 | 弹窗关闭后焦点未回到触发元素 | 中 |
| 键盘导航 | 搜索框、按钮、选择器可 Tab 导航，表格行仅点击 | 中 |

---

## 6. 亮点

- **设计系统完整**：CSS 自定义属性 + 语义化颜色 + 组件分层，具备工程化特质
- **生命周期时间线**：`.lc-timeline` 组件直观展示 Alpha 状态流转
- **评分可视化**：三层评分卡片（先验/实证/清单）+ 进度条 + 决策带，信息密度高但可读
- **Toast 通知链**：带动画、自动消失、点击关闭，四种状态类型
- **进度追踪**：云端同步、批量检查均有实时进度条 + ETA
- **响应式布局**：三档断点（1380/1120/720px），基本覆盖桌面到平板
- **搜索 + 过滤**：表格内搜索实时过滤，支持排序和快捷键

---

## 7. 优先建议

### P0 — 上线前必须修复

1. **批量提交失败明细** — 展开 `results` 逐条展示失败 Alpha + 原因，取消 `ok: true` 时直接切"已提交"的行为
2. **BLOCKED 状态入失败视图** — `failedRows()` 增加 `blocked` 匹配，确保提交被阻断的 Alpha 在正确视图中可见
3. **状态码全量中文化** — 补全 `statusLabelFromCode()` 缺失映射（`duplicate_expression_skipped`, `previously_rejected_expression_skipped`, `simulation_poll_failed`, `simulation_timeout`, `fusion_pending`, `BRAIN_CHECK_FAILED`）
4. **检查结果持久化恢复** — 增加 `/api/check_results` 接口读取 `checks.jsonl`，页面初始化时恢复 `checkResults`

### P1 — QA 期间优先

5. **特殊 Alpha Type 入口** — 下拉增加 Power Pool / ATOM / Pyramid 选项
6. **检查失败原因人性化** — `humanCheckName()` 补全所有检查名，增加"建议操作"字段
7. **事件中心视图** — 在生命周期 Tab 旁增加事件日志 Tab

### P2 — 正式版前

8. **Chart.js 本地化** — 离线可用
9. **无障碍完善** — `role`/`aria-live`/焦点管理
10. **移动端大表格优化** — 卡片视图切换

---

## 8. 评审结论

| 维度 | 评分 | 说明 |
|------|------|------|
| 主路径覆盖 | 🟢 良好 | 生产→回测→达标→检查→提交 全链路可追踪 |
| 非主路径覆盖 | 🔴 不足 | 失败反馈、阻断诊断、特殊类型入口不完整 |
| 视觉设计 | 🟢 良好 | 设计系统完整、层级清晰、语义一致 |
| 可用性 | 🟡 基本可用 | 内部用户可操作，普通用户会在异常路径困惑 |
| 无障碍 | 🟡 基础 | 色彩对比度过关，缺 ARIA/焦点管理 |
| 响应式 | 🟡 基本可用 | 三档断点，移动端大表格体验差 |

**总体判断**：有条件通过进入 QA，不建议直接正式上线。优先修复 P0 四项后再进入 QA 阶段。
